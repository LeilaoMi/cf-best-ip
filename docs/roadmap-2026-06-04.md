# cf-best-ip 阶段推进计划（基于 2026-06-04 审计）

> 来源：`docs/audit-2026-06-04.md`  
> 目标版本建议：`v3.6.1` → `v3.7.0` → `v3.8.0`  
> 原则：先防误操作和滥用，再提升数据可信度，最后拆模块和产品化。每阶段都必须可验证、可回滚、可单独发布。

---

## 0. 总体策略

这个项目当前已经是完整的 Cloudflare Worker 优选 IP 分发系统，不建议继续优先堆数据源。接下来按 5 条主线推进：

1. **安全止血**：先修可能导致配置写坏、管理 token 泄露、线上 stack 暴露的问题。
2. **线上保护**：降低公开接口被抓、刷新接口被刷、DNS 被误同步的概率。
3. **数据可信**：明确哪些 IP 是真实测速、哪些只是来源推荐，避免页面误导。
4. **可观测**：失败时能看到原因、阶段耗时、Cloudflare API 原始错误、通知状态。
5. **可维护**：在继续加功能前拆出纯逻辑模块，并补真正单元测试。

## 2. 阶段 1：安全止血 + 防误操作（`v3.6.1`）

### 目标

先解决最可能造成线上事故的点：配置写坏、URL token 泄露、错误堆栈暴露、健康检查信息不足、README 漂移。

### 任务清单

| 顺序 | 任务 | 来源 | 涉及文件 | 验收标准 |
|---:|---|---|---|---|
| 1 | 给 `/api/config` POST 增加类型校验、范围限制、危险项二次确认 | P0-1 | `src/worker.js` | 非法数字、字符串、数组对象不能写入 KV；危险开关必须带确认字段 |
| 2 | `/api/config` GET 默认返回脱敏配置，raw 模式需要额外确认 header | P0-2 | `src/worker.js` | 默认不暴露检测 API、黑名单等完整策略；raw 需要 admin + header |
| 3 | 管理接口禁用 `?token=`，只接受 `Authorization: Bearer` | P0-3 | `src/worker.js`, README | `/admin?token=xxx`、`/api/config?token=xxx` 不再通过 |
| 4 | 全局 500 不返回 `stack` | P0-8 | `src/worker.js` | 公开响应只返回 `internal-error` + `requestId`；详细错误进 `console.error` 或 KV |
| 5 | `/health` 增加 `status`、`reasons`、`lastErrorAt` | P4-1 | `src/worker.js` | health 能区分 `ok/degraded/down`，并给出原因数组 |
| 6 | README 修正 cron 和 worker 行数漂移 | P5-1/P5-2 | `README.md` | README cron 与 `wrangler.toml` 一致，worker 行数不再写死或更新为当前值 |
| 7 | verify 脚本增加关键漂移检查 | P6-6 | `scripts/verify-worker.mjs` | 能检查 cron、版本、README 关键接口说明 |

### 建议实现顺序

1. 先写 `sanitizeConfigPatch()`，只做白名单字段处理，不改其它业务逻辑。
2. 再改鉴权入口，把 query token 限制在极少数兼容接口，管理接口彻底禁用。
3. 再改错误响应和 health 输出。
4. 最后同步 README 与 verify。

### 必跑验证

```bash
node -c src/worker.js
node scripts/verify-worker.mjs
```

如新增测试框架前暂时没有单元测试，至少要用本地脚本或 curl 模拟：

```bash
# 非法配置应返回 400，不应写入
curl -i -X POST "$BASE/api/config" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{"probeConcurrency":999999}'

# query token 管理接口应失败
curl -i "$BASE/api/config?token=$ADMIN_TOKEN"

# health 应包含 status/reasons
curl -fsS "$BASE/health" | jq '{ok,status,reasons}'
```

### 回滚方案

- 只改 Worker 和 README，不涉及 DNS 结构变化。
- 如果 `/api/config` 校验影响现有配置，可临时保留旧 config 读取逻辑，只限制新写入。
- 如果 query token 禁用影响旧脚本，先只对 `/admin`、`/api/config`、`/api/diagnostics` 禁用，`/api/refresh` 保留一版兼容并返回 deprecation warning。

## 3. 阶段 2：数据可信度和健康状态（`v3.6.2`）

### 目标

让用户明确知道：哪些 IP 是真实测速、哪些是来源推荐；让系统在核心数据源异常时进入 degraded，而不是简单 ok。

### 任务清单

| 顺序 | 任务 | 来源 | 涉及文件 | 验收标准 |
|---:|---|---|---|---|
| 1 | `enrichGeo` 后再次执行 `countryBlocklist` | P2-7 | `src/worker.js` | 来源没带 country 的 CN IP 补全后仍会被黑名单过滤 |
| 2 | 每条 IP 增加 `quality.testedBy`、`quality.confidence` | P2-1 | `src/worker.js` | `/api/ips` 能看出 `hostmonit`、`client-estimated`、`source-only` |
| 3 | 首页表格增加“测速来源/可信度”说明 | P2-1/P2-2 | `src/worker.js` | UI 明确 `ms*` 只是浏览器连接耗时，不等于业务可用 |
| 4 | `qualityGuard` 加 tested 比例、核心源质量指标 | P2-3 | `src/worker.js` | tested 比例大跌或 hostmonit 异常时不自动大换血 DNS |
| 5 | `sourceHealth` 支持 `critical` 和 `weight` | P2-4 | `src/worker.js` | hostmonit 挂掉时 health 为 degraded |
| 6 | `aliasOf` 源统计去重 | P2-5 | `src/worker.js` | 同上游源不会被当成多个独立信号夸大健康度 |

### 必跑验证

```bash
node -c src/worker.js
node scripts/verify-worker.mjs
curl -fsS "$BASE/health" | jq .
curl -fsS "$BASE/api/ips?top=5" | jq '.[0] // .items[0]'
```

### 注意事项

- 不要把“客户端临时测速”包装成“Worker 实测”。Cloudflare Worker 不能做真实 TCP 业务测速，页面必须写清楚。
- `qualityGuard` 先保守，不要一次性拦截太多正常刷新；建议只在明显异常时降级。

## 4. 阶段 3：降滥用 + 降资源消耗（`v3.6.3`）

### 目标

减少公开接口重复抓取、移动端测速负担、Cloudflare API 压力和通知静默失败。

### 任务清单

| 顺序 | 任务 | 来源 | 涉及文件 | 验收标准 |
|---:|---|---|---|---|
| 1 | `/sub` 增加 `Cache-Control: public, max-age=300` | P1-3 | `src/worker.js` | 订阅接口可缓存，管理接口仍 `no-store` |
| 2 | 公开 `/api/ips` 默认限制 top 数量 | P1-3 | `src/worker.js`, README | 默认不一次性吐全量；需要全量时有明确参数或轻量 token 策略 |
| 3 | 首页客户端测速只测前 10 个，移动端低并发 | P2-2/P3-4 | `src/worker.js` | 移动端并发 2-3，页面不可见时暂停 |
| 4 | `ALLOW_PUBLIC_REFRESH=1` 时首页和 stats 红色警告 | P1-4 | `src/worker.js`, README | 线上一眼能看出高危开关开启 |
| 5 | DNS 同步优先按 managed name 查询 | P0-6 | `src/worker.js` | managed names 少时不再全量扫整个 zone A 记录 |
| 6 | Telegram 通知检查 `r.ok`，失败写入 KV | P1-6 | `src/worker.js` | `/api/diagnostics` 可看到最近通知错误 |
| 7 | `ip-api.com` HTTP fallback 改为不参与决策或移除 | P1-5 | `src/worker.js` | HTTP 数据不会影响国家黑名单和 DNS 同步 |

### 必跑验证

```bash
node -c src/worker.js
node scripts/verify-worker.mjs
curl -I "$BASE/sub"
curl -fsS "$BASE/api/stats" | jq .
```

### 注意事项

- `/api/ips` 是否默认只返回 top 20，需要和 README 同步，避免旧用户脚本突然拿不到全量。
- DNS 查询策略属于线上敏感逻辑，改完必须先 dry-run 或只记录 diff，不要第一次就直接部署同步。

## 5. 阶段 4：拆模块 + 真测试（`v3.7.0`）

### 目标

降低 2068 行单文件维护风险。先拆纯逻辑，确保行为不变，再补 `node --test`。

### 拆分顺序

第一批只拆纯函数，风险最低：

```text
src/
├── worker.js          # 入口和路由，尽量只保留编排
├── config.js          # DEFAULT_CFG / config sanitize / env check
├── cidr.js            # IPv4/IPv6 CIDR 判断
├── scoring.js         # scoreIp / qualityGuard / sourceHealth
├── dns.js             # Cloudflare DNS API / sync diff
└── utils.js           # json/text/html/pMap/timeout 等通用工具
```

第二批再拆模板和数据源：

```text
src/
├── sources.js         # SOURCES / fetchSource / aggregateSources
├── geo.js             # enrichGeo / risk
├── render-home.js     # 首页 HTML
└── render-admin.js    # admin HTML
```

### 任务清单

| 顺序 | 任务 | 来源 | 验收标准 |
|---:|---|---|---|
| 1 | 抽出 `cidr.js` 并导出 `isCfNativeIp` 等纯函数 | P3-1/P6-2 | 行为不变，CIDR 边界测试通过 |
| 2 | 抽出 `scoring.js` | P3-1/P6-1 | `qualityGuard` 测试覆盖缩水、核心源失败、tested 比例 |
| 3 | 抽出 `dns.js` | P3-1/P6-1 | DNS diff、maxChanges、保留旧 IP、补满 topN 测试通过 |
| 4 | 增加 `node --test` 单元测试 | P6-1/P6-7 | workflow 可跑 tests，不触碰真实 Cloudflare |
| 5 | workflow 增加 syntax + verify + unit tests | P6-7 | PR/推送自动验证 |
| 6 | 拆 `render-home.js`、`render-admin.js` | P6-3 | 页面行为不变，HTML 生成函数独立 |

### 必跑验证

```bash
node -c src/worker.js
node scripts/verify-worker.mjs
node --test
```

如项目改为 ESM 多文件，还要确认 Wrangler 能正常打包：

```bash
npx wrangler deploy --dry-run
```

### 注意事项

- 这一阶段禁止顺手改业务策略。拆模块就是拆模块，不能混入功能变化。
- 每次只拆一个模块，一个 commit 只做一件事。
- 拆完每一步都跑验证，避免最后一次性爆炸。

## 7. 暂缓项

这些不是不做，而是不建议排在前面：

1. **CSP nonce 化**：有价值，但当前 HTML/JS 内联较多，改动大；建议等模板拆分后做。
2. **Durable Object 强锁**：最稳，但会增加部署资源和复杂度；短期先用 `refresh:running` + 冷却。
3. **完整 host 白名单和 ROOT_DOMAIN 重构**：能防误绑定，但涉及部署变量和 DNS 角色，建议单独小版本做，并先写配置检查。
4. **AAAA 自动同步**：IPv6 已有数据，但默认同步 AAAA 可能让部分客户端故障，必须默认关闭。
5. **公开接口强鉴权**：会影响订阅使用体验，先用缓存、默认 top 限制和文档提示降风险。

## 8. 每阶段统一验收门槛

每个阶段完成前，必须满足：

```bash
node -c src/worker.js
node scripts/verify-worker.mjs
```

如果阶段 4 后已有单元测试，则追加：

```bash
node --test
```

上线前必须确认：

```bash
curl -fsS --max-time 20 https://bestip.leilaomi.cc.cd/health | jq .
curl -fsS --max-time 20 https://bestip.leilaomi.cc.cd/api/stats | jq .
```

并检查：

- `/health` 为 `ok` 或明确可接受的 `degraded`。
- `sourceHealth` 没有核心源全挂。
- DNS 同步没有异常大规模删除。
- README 与实际 cron、版本、接口参数一致。
- 没有把任何 token/key 写进仓库或响应体。

## 10. 最小可落地路径

如果只想最快拿到明显收益，按这个 10 项做：

1. `/api/config` 类型校验。
2. 管理接口禁用 query token。
3. 500 不返回 stack。
4. README 修正 cron 和行数漂移。
5. `enrichGeo` 后再次执行 countryBlocklist。
6. `/health` 返回 reasons 和 degraded 状态。
7. `sourceHealth` 给 hostmonit 加 critical 权重。
8. 首页/API 明确 `testedBy/confidence`。
9. 客户端测速降并发、只测前 10 个。
10. 抽出 DNS diff 逻辑并补单元测试。

这 10 项完成后，项目的安全性、可信度、可维护性都会有明显提升，而且不会引入“再加更多源”那种高风险复杂度。
