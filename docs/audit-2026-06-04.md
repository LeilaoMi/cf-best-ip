# cf-best-ip 项目改进建议全册（严谨版）

> 审计日期：2026-06-04  
> 审计对象：`LeilaoMi/cf-best-ip` 本地 `main`，远端 `origin/main` 已校准一致  
> 当前版本：`v3.6.0`，最新提交 `d9dc11c`  
> 核心文件：`src/worker.js` 2068 行、`wrangler.toml`、`.github/workflows/*`、`scripts/verify-worker.mjs`  
> 审计方式：逐段阅读关键代码 + `node -c` 语法检查 + 项目自带 verify 脚本 + 线上 `/health` 探活 + 静态 grep 定位。

---

## 0. 先说结论

这个项目已经不是“简单优选 IP 列表”，而是一个完整的 **Cloudflare Worker 优选 IP 分发系统**：多源聚合、CF CIDR 校验、KV 持久化、DNS diff 同步、三网分流、管理页、健康检查、GitHub Actions 监控都已有。

真正值得继续优化的方向不是“再堆更多源”，而是下面 5 件事：

1. **防误操作**：避免 `/api/config`、DNS 同步、域名推断把线上记录写坏。
2. **降滥用风险**：减少公开页面和接口被爬、被刷、被 Cloudflare 风控误判的机会。
3. **增强真实性**：现在多数 IP 不是 Worker 自测，而是依赖第三方源；要把“可信度”和“测速来源”讲清楚。
4. **增强可观测性**：失败时直接告诉你“为什么失败、哪一步失败、该查什么”，而不是只给 `ok:false`。
5. **降低维护成本**：2068 行单文件已经到维护临界点，继续加功能前应该先拆模块、补真实测试。

# P0 · 可能导致线上误写、误开放、错误使用

## P0-1. `/api/config` 允许直接写运行时配置，但没有类型校验和危险值拦截

**位置**：`src/worker.js` `/api/config`，约 1487-1511 行。

**现状**：

```js
const allowed = Object.keys(DEFAULT_CFG);
const patch = {};
for (const k of allowed) {
  if (body[k] !== undefined) patch[k] = body[k];
}
const next = { ...current, ...patch };
await kvSet(env, "config", next);
```

**风险**：

- `probeConcurrency` 可被写成超大值，导致 cron 抓源/检测过载。
- `availabilityCheckConcurrency` 可被写成超大值，烧第三方 API 或触发风控。
- `dnsBlocklist` 可被写成非数组，后续逻辑表现不可控。
- `perCountryTopN`、`topN` 等字段可被写成负数、字符串、异常对象。

**建议**：

加一个 `sanitizeConfigPatch()`：

- 布尔值只接受 boolean 或 `"1"/"0"`。
- 数字字段限制范围，例如 concurrency 1-30、timeout 1000-15000。
- 数组字段只接受字符串数组，国家代码统一大写且必须两位。
- 写入前返回 `changedKeys`，管理页显示本次改了什么。
- 对危险项（开启 availability/risk filter）要求二次确认字段，如 `confirm: "I_UNDERSTAND"`。

---

## P0-2. `/api/config` GET 会返回完整运行时配置，信息披露过多

**位置**：`src/worker.js` `/api/config` GET。

**现状**：需要 `ADMIN_TOKEN`，但一旦 token 泄漏，会返回 `availabilityCheckApi`、`riskCheckApi`、黑名单、检测开关等完整策略。

**风险**：

- 暴露你如何筛选 IP、屏蔽哪些国家、是否启用风险检测。
- 被别人复刻策略或针对性绕过。
- 如果未来把敏感 URL/API key 误放进 config，会被直接返回。

**建议**：

- 默认 GET 返回脱敏版。
- 提供 `?raw=1` 时才返回完整 config，并要求 `ADMIN_TOKEN` + 额外确认 header。
- 明确禁止把 token/key 写入 KV config，只允许进 Cloudflare secrets。

---

## P0-3. 管理鉴权仍接受 URL `?token=`，容易泄露

**位置**：`queryToken()` / `requestToken()`，约 342-347 行。

**现状**：

```js
function requestToken(request) {
  return bearerToken(request) || queryToken(request);
}
```

虽然管理登录页已经不用 `/admin?token=`，但后端仍接受 query token。

**风险**：

- token 出现在浏览器历史、反向代理日志、截图、分享链接。
- 如果用户从旧文档/旧习惯访问 `/admin?token=xxx`，仍会被接受。

**建议**：

- `/admin`、`/api/config`、`/api/diagnostics` 禁止 query token。
- 如果要兼容旧客户端，仅允许 `/api/refresh?token=` 临时保留一版，并在响应中返回 deprecation warning。
- 最终统一只接受 `Authorization: Bearer ...`。

---

## P0-4. DNS 同步使用 `CF_RECORD_NAME` 推断根域，配置错会派生错误子域

**位置**：`getRootFromRecord()` / `getManagedDnsNames()`，约 1079-1102 行。

**现状**：

```js
function getRootFromRecord(recordName) {
  return recordName && recordName.includes(".") ? recordName.split(".").slice(1).join(".") : "";
}
```

如果 `CF_RECORD_NAME=cf.bestip.example.com`，root 会被推成 `bestip.example.com`，然后派生：

- `ct.bestip.example.com`
- `cu.bestip.example.com`
- `cm.bestip.example.com`

这未必是用户想要的。

**风险**：DNS 写到错误子域，排查困难。

**建议**：

- 新增显式 `ROOT_DOMAIN`，例如 `leilaomi.cc.cd`。
- `ct/cu/cm/auto/cf` 全部基于 `ROOT_DOMAIN` 或显式 record name，不再从 `CF_RECORD_NAME` 猜。
- 启动时校验：所有 managed DNS name 必须以 `.${ROOT_DOMAIN}` 结尾。

---

## P0-5. `SERVICE_HOSTNAME` 只用于展示，没有校验当前请求 host

**位置**：`handle()` 与 `renderHome()`。

**现状**：用户访问任何绑定到 Worker 的 host，页面仍会按 `SERVICE_HOSTNAME` 展示推荐入口。

**风险**：

- 如果误把优选池域名绑定到了 Worker，页面仍正常展示，用户不容易发现域名角色混乱。
- 搜索引擎/爬虫可能通过非预期 host 访问同一页面。

**建议**：

- 对页面路由 `/`、`/admin`、API 入口加入 host 白名单。
- 如果 `request.url.host !== SERVICE_HOSTNAME`，返回 421/400，并提示“该域名不是管理入口”。
- 对 `/health` 可放宽，便于监控。

---

## P0-6. DNS 同步每次列出整个 Zone 的 A 记录，Zone 大时可能触发 API 压力

**位置**：`syncAllDns()` 调用 `listAllARecords()`，约 1195 行。

**现状**：先拉全 zone 所有 A 记录，再筛 managed names。

**优点**：减少多个子域分别请求的复杂度。

**风险**：如果 zone 中 A 记录很多，每 6 小时全量分页会变重；未来你其它项目也放同一个 zone 时，API 压力上升。

**建议**：

- managed names 少时，优先按 name 分别查询。
- managed names 多于一定数量时才全量拉取。
- 记录 `cfApiRequests` 到 `dns:lastSync`，知道一次同步花了多少 API 请求。

---

## P0-7. 手动刷新冷却是 KV 读后写，不是强原子锁

**位置**：`/api/refresh`，约 1514-1524 行。

**现状**：

```js
const prevRaw = await env.KV?.get("refresh:cooldown");
...
await env.KV?.put("refresh:cooldown", String(Date.now()), { expirationTtl: 120 });
```

**风险**：并发请求同时读到旧值时，可能一起进入刷新流程。

**建议**：

- 简单方案：把冷却拉长到 5 分钟，并在响应中明确刷新中。
- 稳妥方案：用 Durable Object 做单实例锁。
- 折中方案：KV 增加 `refresh:running`，刷新结束再释放；即使不是强原子，也能挡住多数重复请求。

---

## P0-8. 全局 500 响应返回 `stack`，线上不该暴露堆栈

**位置**：入口 catch，约 1573 行。

**现状**：

```js
catch (e) { return json({ ok: false, error: String(...), stack: e?.stack }, { status: 500 }); }
```

**风险**：暴露内部函数名、代码结构、异常路径。

**建议**：

- 公开响应只返回 `{ ok:false, error:"internal-error", requestId }`。
- 详细错误写入 KV `runtime:lastError` 或 `console.error`。
- `/api/diagnostics` 管理接口中再查看详细错误。

# P1 · 安全与滥用风险

## P1-1. CSP 仍依赖 `'unsafe-inline'`

**位置**：`SECURITY_HEADERS`，约 262-266 行。

**现状**：

```js
"content-security-policy": "default-src 'self'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src * data:; connect-src 'self'"
```

**风险**：有 CSP 但实际 XSS 防护能力有限。

**建议**：

- 短期：保留 inline style，但脚本改为 nonce。
- 中期：把 JS 拆成独立 `/assets/app.js` 或单独路由返回 JS。
- 限制 `img-src` 不要 `*`，至少明确 `https:`。

---

## P1-2. 管理 token 存在 `sessionStorage`

**位置**：`renderAdminLogin()` 与 `renderAdmin()`。

**现状**：`ADMIN_TOKEN`、`REFRESH_TOKEN` 存浏览器 `sessionStorage`。

**风险**：如果页面被注入脚本，token 可被读取。

**建议**：

- 短期：只存在 JS 变量，不落 storage；刷新页面重新输入。
- 中期：实现短期 session cookie（HttpOnly + SameSite + Secure），服务端校验签名。
- 对这个项目而言，短期方案足够简单安全。

---

## P1-3. 公开 `/api/ips`、`/sub` 可被批量抓取

**位置**：公开接口。

**现状**：默认公开全部 IP 池。

**风险**：

- 被爬虫高频抓取，增加 Worker 请求量。
- 被外部直接复用，可能放大 Cloudflare ToS/滥用风险。

**建议**：

- 默认公开只返回 top 20。
- 超过 top 50 需要轻量 token 或 `?public=1` 限制。
- `/sub` 加 `Cache-Control: public, max-age=300`，降低重复请求压力；但管理/API 仍 no-store。
- 为 `/sub` 增加简单 User-Agent 黑名单/限速提示。

---

## P1-4. `ALLOW_PUBLIC_REFRESH=1` 是高危开关，应强提示

**位置**：`requireRefreshAuth()`。

**现状**：设置后任何人都能 POST 刷新。

**风险**：烧第三方源、Cloudflare DNS API、KV 写入、触发风控。

**建议**：

- 若启用，首页直接显示红色警告。
- `/api/stats` 标记 `publicRefreshEnabled: true`。
- README 中把它写成“仅临时调试，不建议线上”。

---

## P1-5. `ip-api.com` HTTP fallback 仍可能污染地理数据

**位置**：`enrichGeo()`，约 874-905 行。

**现状**：主用 `https://ipwho.is/batch`，失败后 fallback 到 `http://ip-api.com/batch`。

**风险**：HTTP 明文返回被中间人篡改时，country 可能被污染，影响黑名单和 DNS 同步。

**建议**：

- 优先换成 HTTPS fallback，例如 `https://ipapi.co`、`https://api.iplocation.net`，或直接没有 fallback。
- 如果继续用 HTTP fallback，仅用于展示，不参与 DNS 黑名单决策。
- 在 item 上标记 `geoSource` 与 `geoTrusted`。

---

## P1-6. Telegram 通知未检查发送结果

**位置**：`notify()`。

**现状**：`fetch()` 后不检查 `r.ok`。

**风险**：通知失效时你不知道。

**建议**：

- 检查 Telegram API 返回。
- 失败写入 `notify:lastError`。
- `/api/diagnostics` 展示最近通知状态。

---

## P1-7. Cloudflare API token 最小权限应写得更严格

**位置**：README / wrangler 注释。

**现状**：写了 Zone:DNS:Edit。

**建议**：

README 增加明确说明：

- 权限只给目标 zone。
- 只需要 DNS Edit，不要给 Account 全局权限。
- 不要把 API token 写进 `wrangler.toml`，只能 secret。

---

## P1-8. robots.txt 只能减少索引，不能防爬

**位置**：`/robots.txt`。

**现状**：Disallow admin/API/sub。

**建议**：文档里明确“robots 不是访问控制”，真正敏感接口必须鉴权。

# P2 · 数据真实性与可用性

## P2-1. “测速”大多不是本项目实测，应在 UI 明确区分

**位置**：`runFullTest()` 与首页表格。

**现状**：hostmonit 有真实延迟/丢包/速度；其它源大多没有，页面再用浏览器尝试客户端测速。

**风险**：用户以为所有 IP 都被本项目真实验证过。

**建议**：

- 表格增加“测速来源”列：`hostmonit 实测` / `客户端临时测` / `来源推荐未测`。
- `/api/ips` 每条加 `quality: { testedBy, confidence }`。
- 首页说明“Worker 不做 TCP 测速，原因是平台限制”。

---

## P2-2. 客户端测速直接访问 `https://IP/cdn-cgi/trace`，结果不一定代表可用

**位置**：`clientTestIp()`，约 1945-1960 行。

**现状**：用 `<img>` 访问 `https://IP/cdn-cgi/trace`，`onerror` 也算成功，测的是连接耗时近似值。

**风险**：

- TLS 证书/SNI 不匹配时仍可能 onerror，但未必代表该 IP 能稳定作为目标业务的 Cloudflare 入口。
- 移动端大量并发测 IP 会耗电/耗流量。

**建议**：

- UI 标注 `ms*` 的含义：仅浏览器连接耗时，不等于业务可用。
- 默认只测当前可见前 10 个，用户点击“测更多”再继续。
- 更高级方案：维护 `probe.example.com`，让每个候选 IP 通过同一 SNI/Host 测 `/cdn-cgi/trace`。

---

## P2-3. `qualityGuard` 只看数量，不看质量

**位置**：`qualityGuard()`，约 659-672 行。

**现状**：如果总池数量或三网数量缩水才拦截。

**风险**：数量没变但质量变差（延迟变高、hostmonit 源全没了、tested 比例下降）仍会同步 DNS。

**建议**：

加入质量指标：

- tested IP 数量不能低于上一批 50%。
- hostmonit 三网源失败时，三网 DNS 不自动大换血。
- top10 平均 delay 比上一批高 2 倍时，进入 degraded 模式。

---

## P2-4. `sourceHealth` 只看源数量，不区分核心源和普通源

**位置**：`sourceHealth()` 与 `/health`。

**现状**：17/18 正常即可 ok。

**风险**：如果挂的是 hostmonit，影响远大于挂一个普通 raw 源。

**建议**：

- 给数据源增加 `weight` 和 `critical`。
- `/health` 返回 `criticalSourcesOk`。
- hostmonit 失败但其它源正常时，状态应是 `degraded`，不是简单 ok。

---

## P2-5. 数据源 `aliasOf` 目前只标记，未真正用于统计去重

**位置**：`SOURCES` 中 `CMLiussss/cm`。

**现状**：`aliasOf` 存在，但 `sourceStats` 仍逐源统计。

**建议**：

- 统计页把 alias 源合并展示。
- `sources` 数量计算时，上游相同的源只算 1 个“独立信号”。

---

## P2-6. IPv6 已接入源，但 DNS 仍只同步 A 记录

**位置**：IPv6 源与 `listRecords(type=A)`。

**现状**：可解析/展示 IPv6，但 DNS 同步只写 A，不写 AAAA。

**建议**：

- 如果要真正支持 IPv6，加 `IPV6_RECORD_NAME` 或 `CF_SYNC_AAAA=1`。
- 默认不开启 AAAA，避免客户端不支持导致故障。
- README 明确：当前 IPv6 主要用于订阅/API，不参与默认 DNS 池。

---

## P2-7. 国家黑名单发生在 enrichGeo 之前，已知 country 和补全 country 的处理顺序不一致

**位置**：`runFullTest()`，约 932-938 行。

**现状**：先按已有 country 过滤，再 enrichGeo。

**风险**：来源没带 country 的 CN IP 会先保留，补全后不会再次经过 `countryBlocklist`。

**建议**：

- enrichGeo 后再执行一次 countryBlocklist。
- 或把过滤顺序改成：聚合 → enrichGeo → 国家过滤 → 可用性检测 → 稳定分。

# P3 · 性能与资源消耗

## P3-1. 2068 行单文件会增加维护和冷启动负担

**位置**：`src/worker.js`。

**建议拆分**：

```text
src/
├── worker.js          # 入口和路由
├── config.js          # DEFAULT_CFG / env 校验
├── sources.js         # SOURCES / fetchSource / aggregateSources
├── cidr.js            # IPv4/IPv6 CIDR 判断
├── geo.js             # enrichGeo / risk
├── dns.js             # Cloudflare DNS API / sync
├── scoring.js         # scoreIp / qualityGuard
├── render-home.js     # 首页模板
├── render-admin.js    # admin 模板
└── utils.js           # json/text/html/pMap/timeout
```

先拆不改逻辑，拆完跑 `node -c` 和 verify。

---

## P3-2. 每次请求都读 KV config + latest

**位置**：`handle()` 开头。

**现状**：任何路径都会 `getLatest()` + `getConfig()`，包括 `/robots.txt`、OPTIONS、404。

**建议**：

- OPTIONS、robots、静态轻路由提前返回。
- `/health` 只读必要 KV。
- 首页/API 再读完整 latest。

---

## P3-3. 首页 HTML 大且包含全部 CSS/JS，无法浏览器缓存

**位置**：`renderHome()`。

**现状**：CSS/JS 内联，响应 no-store。

**建议**：

- 首页 HTML 可 no-store，但 CSS/JS 可拆成 `/assets/app.css`、`/assets/app.js` 并缓存。
- 如果不想多文件，至少压缩 CSS/JS 或减少重复内联。

---

## P3-4. 客户端测速默认并发 6，移动端可能偏重

**位置**：`CLIENT_TEST_CONCURRENCY = 6`。

**建议**：

- 移动端降到 2-3。
- 页面不可见时暂停测速：`document.visibilityState`。
- 只测当前 tab 前 10 个。

---

## P3-5. `existingContents.includes()` 在 DNS 同步中可改为 Set

**位置**：`syncRecordFromExisting()`。

**现状**：循环中多次 `includes()`。

**建议**：构造 `existingSet`，微优化即可，不紧急。

---

## P3-6. 风险检测和可用性检测开启后可能超过 Worker 执行时间

**位置**：`applyAvailabilityFilter()` / `applyRiskFilter()`。

**建议**：

- 给 runFullTest 加总耗时预算，如 25 秒。
- 检测前计算预计请求数，超过预算就降采样。
- 把 `elapsedMs` 和各阶段耗时写入 stats。

# P4 · 运维观测与排障

## P4-1. `/health` 缺少失败原因字段

**位置**：`/health`。

**现状**：只返回 ok、total、stale、dnsOk、sourceHealth。

**建议**：增加：

```json
{
  "status": "ok|degraded|down",
  "reasons": ["stale", "dns-failed", "critical-source-failed"],
  "nextCronHint": "...",
  "lastErrorAt": 0
}
```

---

## P4-2. `/api/dns/current` 没要求 admin 鉴权

**位置**：`/api/dns/current`。

**现状**：公开返回 managed DNS 当前记录。

**取舍**：这不是密钥，但会公开你 DNS 池当前 IP。

**建议**：

- 如果项目定位公开订阅，可以保留公开。
- 如果想降低爬取风险，应改为 `ADMIN_TOKEN`，首页只展示汇总。

---

## P4-3. DNS history 每天只保留最后一次，不能看日内多次变更

**位置**：`dns:history:${day}`。

**现状**：同一天多次同步会覆盖。

**建议**：

- key 改为 `dns:history:${day}:${hour}` 或写入数组保留最近 20 次。
- `/api/history` 支持 `type=dns` 查看同步历史。

---

## P4-4. IP history 同样每日覆盖，趋势粒度偏粗

**位置**：`ips:history:${day}`。

**建议**：至少保存最近 48 次 cron 快照，便于发现某个源从几点开始异常。

---

## P4-5. 缺少阶段耗时统计

**位置**：`runFullTest()`。

**建议**：记录：

- `fetchSourcesMs`
- `geoMs`
- `availabilityMs`
- `riskMs`
- `dnsSyncMs`
- `notifyMs`

这样排查“为什么 cron 慢”会很快。

---

## P4-6. Cloudflare API 错误没有保留原始 errors 数组

**位置**：`cfApiJson()`。

**现状**：把 errors map 成 message 字符串。

**建议**：

- 把 `code/message` 数组写入 `dns:lastSync.rawErrors`。
- 对用户响应仍保留简洁 message。

---

## P4-7. GitHub Actions 监控只检查健康，不保存趋势 artifact

**位置**：`.github/workflows/health-check.yml`。

**建议**：

- 每次 workflow 输出 health JSON 为 artifact。
- 或把结果追加到 `docs/health-history.json` 不建议，容易污染仓库。
- 更好：失败时发 Telegram/邮件；成功只保留 logs。
